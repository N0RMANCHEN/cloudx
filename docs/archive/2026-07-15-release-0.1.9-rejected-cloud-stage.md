# Cloudx 0.1.9 Rejected Cloud Stage

## Published Evidence

Cloudx `0.1.9` was built and signed from source commit `72e9d6795403d101857fd18591589e2c998fbdf4` during recovery from an unavailable private release key.

- new public-key fingerprint: `SHA256:jJAS0rbecTDcU4QD3JjkJFv/5YpFyAOAtvdaf0fY4j0`
- immutable artifact ref: `release-artifacts/v0.1.9`
- artifact ref SHA: `cac9e227e90acdcb6d49e0dab56dc62618fa4e27`
- local artifact SHA-256: `b8bd4cf42309e0a8330a6c95965d75c78605c2dc014ab7656fcd858d3e23a3c3`
- cloud artifact SHA-256: `7788fe1ba1d018eabc983694c6b9135ba5648955c270a4a4275d3b2ffada47a8`
- manifest SHA-256: `ef404da8b4dd53a598a71745ec47f608e70e8a550e5da308f753be16bf37b690`
- manifest signature SHA-256: `5afa994607697510694eb9fcf057b051e15de25529f84dfeae1fd2917a65a6fc`
- offline bundle SHA-256: `4ae10af2ceca7f64861cf0f8b6781323572c63165da2070a3a35dd6ba8f32917`

Fresh artifact and stable clones initially verified the manifest, both artifact hashes, source commit, self-check contracts, stable signature, and stable manifest digest. The local candidate, whose packaged trust root had been updated, staged `0.1.9` beside active `0.1.8` without activation.

## Rejected Cloud Stage

The cloud candidate correctly rejected the same signed offline bundle before creating `/opt/cloudx/releases/0.1.9`. Investigation found that the repository and local packaged `allowed_signers` files contained the recovered public key, while `cloud/cloudx_cloud/data/allowed_signers` still contained the prior key.

This was a release-construction defect, not a signature bypass or production mutation. The following remained unchanged:

- cloud `current`: signed `0.1.8`
- cloud `previous`: signed `0.1.7`
- local `current`: signed `0.1.8`
- local `previous`: signed `0.1.7`
- local CLIProxyAPI: PID `38189`
- cloud CLIProxyAPI: PID `977036`, restart count `0`
- old cloud importer: PID `133756`, restart count `0`
- `cloudx-cpa-health.timer`: active and waiting on the signed `0.1.8` unit

No endpoint activated `0.1.9`. No systemd unit, release link, gateway, importer, credential, auth file, quarantine state, listener, or active Codex process changed.

## Stable Recovery And Version Advance

The signed `release/stable` ref was immediately restored to the previously accepted `0.1.8` index and signature. Fresh clone verification used the `0.1.8` embedded trust root, and the installed updater again reported current and available `0.1.8` with no update.

The immutable `0.1.9` artifact ref remains as failure evidence and must not be overwritten or activated. Repository development advances to `0.1.10`, updates the cloud-packaged trust root, and adds a regression requiring all three signer files to remain byte-identical before another release is built.
