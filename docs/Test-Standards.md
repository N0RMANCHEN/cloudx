# Test Standards

## Required Layers

- Unit tests cover command parsing, profile state, tunnel lifecycle, import normalization, atomic writes, health redaction, and release verification.
- Local import tests cover tokenless `agentIdentity` capability rejection, strict synthetic Ed25519 PKCS#8 fixtures, preservation of signing metadata, removal of synthetic bearer/task state, distinct deduplication, mode-`0600` write verification, and private-key redaction. Capability tests bind the exact regular executable digest to a bounded sidecar and a live loopback health header, reject a binary-only upstream update, and automatically accept a newly matching runtime without static declarations. External-candidate tests bind official source, patch, toolchain, deterministic bytes, side-by-side staging, distinct activation, zero-connection gating, auth preservation, and automatic launcher rollback.
- Cloud import tests cover the same bounded tokenless `agentIdentity` normalization, source task/synthetic-token removal, distinct runtime/key fingerprints, private mode-`0600` idempotent writes, invalid-key redaction, and rejection before target-state creation when capability evidence is absent. Cloud capability tests bind `cloudx.cloud-cpa-capabilities.v1` to the exact executable digest and configured gateway health header, and reject stale binaries, symlink manifests, invalid probe URLs, and missing live capability advertisements.
- API diagnosis tests cover explicit deactivation, quota exhaustion, transient rate limiting, relogin, access denial, generic pool failure, reset-time normalization, request-body exclusion, successful-response false positives, and preservation of a definitive root cause across later `no auth available` responses.
- HTTP importer stop-transaction tests cover non-authorizing planning, exact confirmation before file/network access, fresh evidence/digest binding, exact signed remote-gate parity, complete rollback-manifest requirements, active/listener baseline, partial-disable recovery, post-stop canary recovery, restore failure reporting, real SSH dry-run result semantics, listener closure, continuity, and retention of every retirement artifact.
- Legacy local removal tests cover offline non-authorizing planning, exact confirmation before home/state access, exact active-release binding, private recovery path and digest verification, bounded non-symlink runtime inventory, user-private transaction locking, legacy process/listener rejection, pre-move standalone recovery-script creation and compilation, three-target same-filesystem quarantine, fresh-shell/native-import acceptance, external CPA continuity, partial-move rollback, and automatic restoration after any post-move failure.
- Stale local exec retirement tests cover offline planning, confirmation-before-inspection, digest rebinding, exact orphan/child structure, age/TTY/stdio/socket/CPU eligibility, target-count bounds, `SIGTERM`-only behavior, refusal to escalate to `SIGKILL`, and unchanged external CPA identity.
- Legacy local control migration tests cover offline planning, confirmation-before-inspection, five-sample zero-connection and thirty-day idle gates, retained-runtime verification, digest rebinding, private backup permissions, generated recovery-script compilation, one-service restart, authentication canary, CPA/selector continuity, and automatic live-control restoration after any target failure.
- Legacy local control retirement tests cover offline non-authorization, confirmation-before-inspection, retained-runtime and migration-backup binding, repeated zero-connection/idle gates, pre-stop recovery-script compilation, LaunchAgent-only unload/quarantine, port closure, no `SIGKILL` or official Codex termination, CPA/selector continuity, and automatic retained-service restoration on post-stop failure.
- Cloud legacy runtime quarantine tests cover offline non-authorization, confirmation-before-inspection, bounded non-writable and non-symlink tree hashing, process/unit/cron dependency gates, dormant unit requirements, signed selector/self-check and gateway/compatibility continuity, full importer rollback-archive verification, root-private backup and generated recovery-script compilation, same-filesystem quarantine, no service or credential mutation, and automatic exact-tree restoration after post-move failure.
- Release workflow key tests cover offline path-free planning, exact confirmation before key/network access, external private-key ownership/mode/symlink guards, committed-root fingerprint matching, clean pushed-main binding, fixed repository/environment/workflow contracts, absent release refs, stdin-only secret transfer, non-publishing dispatch acceptance, unchanged refs, private transaction locking, secret-free receipts, and explicit no-tag failures after any non-rollbackable secret write.
- Governance and unit tests require every cloud-runtime stdout/stderr path and published public document to pass the Phi metadata boundary, including negative fixtures for Task, session, device, lease, approval, local path, transfer content, ContextRequest, LocalAction, and Phi Artifact fields.
- The default verifier evaluates the committed Phi/Cloudx current-and-N-1 evidence across all four release pairs, Cloudx-first and Phi-first upgrades, and both independent rollback directions. A valid audit may report `blocked`; only `compatible` can satisfy the M4A ordering gate.
- The default verifier validates the migration-only legacy health bridge against a strict example, immutable-artifact systemd boundary, and three separate runtime-acceptance gates. Published evidence binds the annotated tag source, immutable artifact-ref commit, manifest digest, and signer fingerprint while allowing repository development to advance independently; publication can satisfy only its own gate, never unit or rollback acceptance. An optional checkout-aware run executes the exact recorded Phi N-1 parser against generated bridge output.
- The bridge rollback regression must build the candidate artifact, keep it independent from the endpoint selector, drive the real Cloudx rollback function in both directions under an isolated release root, and require byte-identical output before and after each selector change.
- Legacy bridge unit-installer tests cover non-authorizing planning, exact confirmation, staged-artifact binding, signed-template extraction, bounded non-symlink inputs, root-owned fixed paths, active legacy rollback availability, inactive/disabled target enforcement, idempotence, systemd verification, daemon reload, secret-free receipts, and full file rollback without any start/enable/stop/disable or selector operation.
- Legacy bridge canary tests require the signed static unit to publish only under its dedicated `/run` directory, mask the production legacy-health directory, retain the primary and old unit states, reject stale/unsafe output, validate the exact legacy contract, clean temporary state after success or failure, and keep the default plan fully non-authorizing.
- Legacy bridge cutover tests require five ordered phases, conservative-primary versus old-exporter identity separation, target validation before every current-timer disable, no state with both timers inactive, complete old-path failure recovery, final primary restoration, exact selector/process continuity, root-only backup preparation, and a default plan with every mutation authorization false.
- Phi consumer credential installer tests cover read-only planning, exact confirmation, staged-artifact binding, fixed path/group contracts, symlink and size rejection, private existing Cloudx credential continuity, overlap-first rotation, secret-free receipts, gateway canary/watch acceptance, and full config/credential rollback after failure.
- CPA policy tests cover exact upstream-commit and every patch-digest binding, path-bounded cloud Agent Identity composition, deterministic candidate bytes, the process-global two-request business maximum, waiting backpressure, control-endpoint exclusion, preservation of fast-service-tier mapping, first-conclusive permanent-auth receipt emission, identity-free aggregate unavailable/available observations, natural trigger emission for both final `auth_unavailable` and all-candidate `model_cooldown`, provisional refreshable-401 retention, zero unsolicited periodic account probes, infrastructure/provider gating, independent high-concurrency incident probing, duplicate-credential deduplication, bounded `deactivated_workspace` classification, weekly/quota/429/network/TLS/DNS/timeout/5xx exclusion from archive authority, trigger retention on infrastructure failure, digest-bound top-level archive, non-symlink bounded reads, manifest rollback, exact restore, network-free receipt-only consumption, separate network-capable sweep consumption, archive-lock availability during network probes, local dual `WatchPaths` plus fallback timing, signed cloud trigger-aware health and failure/sweep templates, health-timer state preservation, distinct watcher and policy confirmations, process-neutral side-by-side staging, baseline-binary preservation, release-matched consumer/producer gating, health/policy/capability canaries, capability-sidecar publication only after live proof, sidecar rollback on failure, real local official-Codex communication before/after activation and recovery, deferred current-turn-safe scheduling, repeated zero-connection gating before launcher mutation, prebuilt private manual recovery artifacts, launchd unload stabilization, bootstrap retry, safe enumerated worker stages, healthy-baseline no-restart idempotence, and an isolated executable offline-to-healthy baseline recovery rehearsal.
- Deferred local watcher-follower tests require an exact accepted policy5 activation receipt, matching candidate identity, passed real communication, and available service before invoking the separately confirmed watcher transaction. Failed or expired policy activation must produce a watcher receipt with no watcher or CPA mutation; accepted watcher activation retains its own backup and automatic rollback.
- The default verifier also evaluates the committed Phi privileged-boundary evidence across the normal interactive CLI, mail-command, and orchestrator Agent surfaces. It derives effective auth-read, import, gateway-mutation, and release-mutation capability from direct tools, runtime identity, privilege elevation, `NoNewPrivileges`, and path masking. A valid audit may report `blocked`; only `secure` can satisfy the M4A privilege gate.
- The failure-semantics verifier covers all nine dependency outcomes, exact Phi file digests and Roadmap status recording from the recorded immutable Git commit, accepted Phi runtime fixtures, compatible release ordering, and the privileged-boundary result. Checkout-aware verification must read the pinned commit objects rather than require an unrelated development worktree to remain at that HEAD. Tests must prove future Phi Roadmap statuses are informational when declared as such, become blockers when evidence explicitly makes them normative, and never weaken the runtime-fixture, compatibility, permission, owner-truth, or no-mutation gates.
- Installer tests require stage-only trust recovery to use a distinct exact confirmation and to leave activation links, shell source, native profiles, legacy backups, services, and processes unchanged.
- Installer regressions prove that a complete native auth/config pair is never reseeded during upgrade, a fully absent pair may use only a valid regular requested seed account, and partial, symlinked, non-regular, or invalid-source state fails before any backup, fetch, stage, or apply mutation.
- Direct-upgrade regressions cover local-only and cloud-only staging/activation, read-only `--check`, suppression of a competing scheduled index fetch, signed stable-index and manifest-digest binding, downgrade rejection, local selector rollback after activation failure, release-matched shell routing, secret-free `cloudx.upgrade.v1` output, and unchanged official Codex/external-CPA ownership.
- Active cloud CPA import tests cover a non-authorizing plan, exact confirmation before stdin read, a 16 MiB bound, empty-pool and empty-watcher-input prerequisites, signed importer dry-run/apply, one-file ownership/mode validation, real response-text and policy acceptance, identity-free available observation with no unavailable trigger, no raw credential persistence, unchanged CPA service identity, watcher-archive observation, and same-filesystem rollback containment of credential plus newly emitted watcher inputs on unclassified failure.
- Active Agent Identity batch-promotion tests cover exact request/count binding, confirmation before stdin read, signed-release/policy/capability preflight, validation that legitimate tokenless records bypass static missing-token archival while malformed keys remain secret-free failures, distinct private file acceptance, baseline-only cohort isolation, repeated real canaries, idempotent signed preview, unchanged service/archive/watcher state, prepared manual recovery, and automatic containment after unexpected exceptions.
- Contract tests validate every example against the versioned shared schema.
- Integration tests exercise local and cloud entrypoints with fake SSH, gateway, Codex, and filesystem boundaries.
- Live canaries are manual and use a non-production local port and shadow cloud paths.

## Safety Requirements

Tests must never depend on port `18317`, the production auth directory, a production API key, or an active systemd unit. Network and process boundaries are injected or redirected to fixtures.

Every regression test must assert public behavior or a persisted contract. Tests that assert implementation details need a written reason.

The default closeout command is:

```bash
./verify.sh
```

Release changes additionally run signature rejection, hash mismatch, downgrade, offline bundle, and rollback tests.
