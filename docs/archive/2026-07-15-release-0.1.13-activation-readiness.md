## Production-Order Rehearsal

The isolated transaction established signed `0.1.12` as the baseline, then used the current-version artifacts for each transition:

1. activate cloud `0.1.12`
2. activate local `0.1.12` with its signed hook
3. invoke current cloud `0.1.12` to activate cloud `0.1.13`
4. invoke current local `0.1.12` to activate local `0.1.13` with the target hook
5. invoke newly current local `0.1.13` for a same-version apply with `--install-shell-hook`
6. repeat the same-version `0.1.13` apply to prove byte idempotence and N-1 preservation

The first local cutover is necessarily executed by `0.1.12` code. It installed the correct target hook but left one leading blank line in the isolated `.zshrc`, with SHA-256 `0871d8cfad0bef983dfb121b440881c823ffb04de871425484a3509465f01b29`.

The immediate same-version apply ran under newly current `0.1.13`, removed that inherited whitespace, and produced canonical `.zshrc` SHA-256 `a552418a59d2b0e194fa12595cb1700c5a80d532ee186cbf0d8dec8b7dea5e74`. A repeated apply retained that exact hash and kept local `previous=0.1.12`.

The production activation procedure must therefore include this one-time post-cutover local reconciliation after the first `0.1.13` apply.

## Source 0.1.14 Alias Fix

An initial rehearsal under macOS `/tmp` exposed a path-alias defect in signed `0.1.13`: `current.resolve()` returned `/private/tmp/...`, while the same-version destination retained `/tmp/...`. The unequal spellings caused a repeated local apply to replace `previous` with the current version.

Canonical production roots `/Users/hirohi/.local/lib/cloudx` and `/opt/cloudx` do not traverse that alias and were not affected. Source `0.1.14` now compares the resolved destination while retaining the configured link target. A regression activates through a symlinked home, repeats the same version, and requires `current=0.1.13` with `previous=0.1.12`.

Full `./verify.sh` passed architecture validation and 123 tests, producing healthy `0.1.14` local and cloud builds.
